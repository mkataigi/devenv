# User Workflow Preferences (Standard Operating Procedures)

新しいタスクを開始する際は、必ず以下のフローに従って進行してください。

## Phase 1: 初期計画と見積もり (Timebox: ~1 min)
ユーザーからの指示を受けたら、直ちに実装を始めず、まず**1分程度**で以下の検討を行ってください。
1. **対応内容の概要**: 何を変更・実装するか。
2. **所要時間の見積もり**: 実装完了までにどの程度の時間がかかるか（待ち時間の予測のため）。

**制約事項:**
- もし計画策定自体が1分で終わらない場合は、そこで一旦思考を打ち切り、「計画の残項目」と「検討終了までの追加所要時間」をユーザーに報告してください。

## Phase 2: ユーザー合意形成
Phase 1の計画を提示した後、**必ずユーザーからのフィードバックを待ってください**。
- ユーザーから修正指示があれば計画を更新します。
- ユーザーの「OK」「進めて」等の合意があるまで実装フェーズ（ファイル変更やコマンド実行）に進まないでください。

## Phase 3: 実装直前の最終確認
実装を開始する直前に、改めて以下を提示してください。
1. **最終的な検討内容**: これから行う作業の確定版。
2. **想定実行時間**: 作業完了までの見込み時間。
3. **承認が必要なコマンドの事前提示**:
   - `file_write`, `bash` など、承認が必要になりそうなコマンドを事前にリストアップし、「これらの実行を承認しますか？」と確認してください。
   - これにより、途中の承認待ちによる中断を防ぎます。

## Phase 4: 作業完了時の報告
作業が完了したら、修正内容を項目ごとにまとめて報告してください。
- 修正概要
- 修正項目ごとの内容詳細（ファイル単位ではなく、論理的な修正内容ごとにまとめる）
